# API仕様ドキュメント運用ルール

## 目的

このルールは、API実装とドキュメントの整合性を維持するための運用方針を定義します。

---

## 基本原則

**実装とドキュメントは常に同期すべきである**

- API実装に影響する変更を行った場合、関連する仕様ドキュメントの更新を **SHOULD（推奨）** とする
- ドキュメント更新が必要か判断に迷う場合は、ownerにレビュー依頼する

**注意:**
- このプロジェクトでは、現時点で実装差分からのドキュメント更新を機械的に強制していません
- これは「運用規範」として位置づけられます
- 将来的にはCIでの自動チェックを導入する可能性があります（拡張案）

---

## 実装変更時のドキュメント更新義務

### 更新が必須の変更

以下の変更を行った場合、**必ず** 関連ドキュメントを更新してください：

#### 1. エンドポイントの追加・削除・変更

**例:**
- 新しいAPIエンドポイントを追加
- 既存エンドポイントのパスを変更
- エンドポイントを廃止

**更新対象:**
- `docs/10_api/endpoints.md`
- 関連する概要ドキュメント

#### 2. リクエスト/レスポンス形式の変更

**例:**
- 必須パラメータの追加・削除
- レスポンスフィールドの追加・削除・型変更
- ステータスコードの変更

**更新対象:**
- 該当エンドポイントのドキュメント
- `docs/10_api/errors.md`（エラーレスポンスの場合）

#### 3. 認証・認可の変更

**例:**
- 認証方式の変更
- 必要な権限（scope/role）の変更
- トークン形式の変更

**更新対象:**
- `docs/00_overview/auth.md`
- 該当エンドポイントのドキュメント

#### 4. エラーハンドリングの変更

**例:**
- 新しいエラーコードの追加
- エラーレスポンス形式の変更
- エラーメッセージの変更

**更新対象:**
- `docs/10_api/errors.md`
- 該当エンドポイントのドキュメント

#### 5. 互換性に影響する変更

**例:**
- 破壊的変更（breaking change）
- 非推奨（deprecated）APIの追加
- バージョニングの変更

**更新対象:**
- 該当エンドポイントのドキュメント
- ADR（Architecture Decision Record）の作成を検討

---

## 軽微変更の判断基準

以下の変更は、ドキュメント更新が **不要** または **任意** です：

### 更新不要

- 内部実装の最適化（外部仕様に影響なし）
- ログ出力の変更
- コメントの追加・修正
- テストコードの追加・修正

### 更新任意（推奨）

- レスポンスフィールドの追加（後方互換性あり）
- エラーメッセージの文言改善（コード変更なし）
- パフォーマンス改善（仕様変更なし）

**判断に迷う場合:**
- ownerに相談
- 「外部から見た挙動が変わるか？」を基準に判断

---

## ドキュメント更新フロー

### 1. 実装変更を行う

- API実装を変更
- テストを追加・更新

### 2. 関連ドキュメントを特定

- 影響範囲を確認
- 更新が必要なドキュメントをリストアップ

### 3. ドキュメントを更新

- 該当ドキュメントを編集
- `status: approved` の場合、必須メタデータを維持

### 4. レビュー

- 実装とドキュメントの両方をレビュー
- ドキュメントの `last_reviewed` を更新

### 5. コミット・PR

- 実装とドキュメントを同じPRに含める
- PRタイトル・説明にドキュメント更新を明記

**推奨PRタイトル例:**
```
feat: Add user profile endpoint with docs
fix: Update auth spec for new token format
```

---

## ドキュメント更新漏れの防止

### ローカル開発

- 実装変更時に関連ドキュメントを同時に開く
- コミット前に `docs/` 配下を確認

### コードレビュー

- レビュアーは実装とドキュメントの整合性を確認
- ドキュメント更新漏れがあればPRをブロック

### CI（将来拡張案）

以下のチェックを導入する可能性があります：

- 実装ファイル変更時に、関連ドキュメントの更新を要求
- OpenAPIスキーマとの自動同期チェック
- エンドポイント一覧の自動生成と差分検出

**注意:**
- 現時点では実装していません
- 導入時は別途ルール文書を作成します

## ADR（Architecture Decision Record）の活用

重要な設計判断を行った場合、ADRを作成することを推奨します。

### ADR作成が推奨されるケース

- 破壊的変更（breaking change）
- 新しいエラーハンドリング方式の導入
- 認証・認可方式の変更
- API設計の大きな方針転換

### ADRの配置

- `docs/30_adr/` 配下に配置
- ファイル名: `adr-NNNN-<title>.md`

**例:**
```
docs/30_adr/adr-0001-error-format.md
docs/30_adr/adr-0002-auth-token-format.md
```

### ADRのメタデータ

ADRも `status: approved` で管理することを推奨します：

```yaml
---
title: ADR-0001: エラーレスポンス形式
owner: platform-team
status: approved
last_reviewed: 2026-03-01
---
```

## OpenAPIとの関係（将来拡張案）

このプロジェクトでは、現時点でOpenAPIスキーマを導入していません。

### 将来的な拡張

- `openapi.yaml` を導入し、エンドポイント定義を機械可読化
- MkDocsとOpenAPIを連携（プラグイン使用）
- 実装とOpenAPIスキーマの自動同期チェック

### 導入時の方針

- OpenAPIは「機械仕様」として扱う
- MkDocsは「人間仕様」として扱う
- 両者を補完関係として運用

**詳細:**
- 導入時は別途ルール文書を作成します

---

## 例外

以下の場合は、このルールの適用外とすることができます：

- 内部API（外部公開しない）
- 実験的なAPI（`status: draft` で管理）
- 一時的なAPI（短期間で廃止予定）

例外を設ける場合は、ドキュメントに明記してください。
